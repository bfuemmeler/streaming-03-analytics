### Kafka Messages

The Kafka producer reads sales records from the
 `data/sales.csv` file and sends each sale record
 as an individual Kafka message.

The producer sends messages to the Kafka topic
configured in the `.env` file. The topic used for
this project was the sales topic created for the
streaming analytics assignment.

Originally, the producer used `region_id` as the
Kafka message key. I modified the producer to use
 `product_id` as the Kafka message key instead.
 This allowed the consumer to compare how different
 payment types appeared in the stream and count
 messages by key.

I also modified the consumer to process selected
message fields instead of storing the entire raw
message. The consumer extracted fields such as:

- message_key
- store_id
- item
- quantity
- price

The consumer then wrote the selected fields to a
CSV output file and maintained running counts of
messages grouped by Kafka key.

These modifications demonstrated how Kafka
consumers can perform real-time analysis and
aggregation on streaming data while messages
are actively being consumed.

### Consumer Processing

The Kafka consumer receives sales messages from the
Kafka topic created for the streaming analytics
project. Each message contains a sales record
sent by the producer from the `sales.csv` file.

The consumer is configured to read messages
from the beginning of the Kafka topic and continues
consuming messages until the configured maximum
number of messages is reached or the timeout period expires.

For each message received from Kafka, the consumer:

- reads the Kafka message key and message value
- processes selected fields from the message
- logs message activity and running counts
- writes selected records to a CSV output file

The consumer logs information during execution, including:

- successful Kafka connection
- topic verification
- consumed message data
- message counts grouped by Kafka key
- total consumed messages

I modified the consumer to track how many messages were
received for each Kafka message key. The consumer
maintained a dictionary of running totals and logged updates such as:

```text
Current key counts: {'credit': 4, 'cash': 2, 'debit': 1}

## Interpretation

This Kafka streaming workflow demonstrated how data can move continuously
between a producer and consumer in real time. Instead of processing a
completed file all at once, the producer sent sales records one message
at a time while the consumer immediately received and processed those messages.

One important change from the original example was modifying the Kafka
message key. Originally, the producer used `region_id` as the message key.
I changed the key to `product_id` so the consumer could group and compare
messages by product instead of geographic region.

I also modified the consumer to:
- process selected message fields
- maintain running message counts by Kafka key
- write filtered records to a CSV file
- log real-time stream summaries

Watching messages move through Kafka helped demonstrate how streaming
systems process live data continuously instead of waiting for batch
uploads. I could see messages appear almost instantly in the consumer
after being produced, which showed how Kafka supports real-time
communication between systems.

The stream could provide useful information to a business or organization
by showing live operational activity as it happens. For example, a company
could monitor:
- buying trends by product type
- sales activity
- transaction volume
- regional performance
- customer behavior patterns

The consumed messages also demonstrated how streaming analytics can
generate business intelligence in real time. By grouping messages
by Kafka key, the consumer was able to identify how frequently certain
payment types appeared in the stream. This type of information
could help businesses:
- monitor sales patterns by product or product category
- identify popular payment methods
- detect unusual activity
- track operational performance
- support faster business decisions

Overall, this project showed how Kafka can be used as a real-time
streaming platform for collecting, processing, analyzing,
and storing live business data.
